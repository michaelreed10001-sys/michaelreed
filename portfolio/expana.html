Project overview

Mintec has provided commodity price intelligence to the world's largest food manufacturers, retailers, and procurement teams for over three decades. Repositioned under a new parent company, the business needed more than a new name — it needed a product experience that matched its elevated ambitions.

The brief: redesign the core commodities analytics platform around Power BI-native components, launch a fully integrated forecasting module, and deliver everything under the new Expana brand — while retaining trust with existing enterprise customers.

This wasn't a cosmetic refresh. The previous Mintec Analytics interface had accumulated years of incremental feature additions — resulting in a dense, inconsistent navigation that was difficult for new users to penetrate. Analysts, procurement strategists, and category buyers needed faster access to price forecasts, historical trends, and custom dashboards. The redesign addressed every layer of the experience.

Mintec → Expana: the rebrand journey

The rebrand wasn't a simple logo swap. Mintec had built significant brand equity in agri-food data over 30+ years. The transition to Expana required careful sequencing — preserving institutional trust while signalling a step-change in capability and product ambition.

Phase 1 — Discovery
Platform audit and user research
Comprehensive heuristic review alongside research sessions with analysts, procurement leads, and category buyers. Identified 23 usability issues across commodity detail, markets, and cost model pages.
Phase 2 — Brand Identity
Expana visual language defined
New colour system anchored in deep forest green (#1A5C3A) and crisp whites. Typography refreshed to Inter. Logo lockup designed to function both standalone and co-branded as "Mintec by Expana" during the customer transition period.
Phase 3 — Core Platform Redesign
Navigation, Home, Commodities, and Markets pages
Full redesign of the primary user journeys. New sidebar navigation system. Home dashboard with personalised commodity feeds and commentary. Commodity detail page with History / Forecast toggle and embedded Power BI chart.
Phase 4 — Power BI Integration
Forecasting and analytics modules
All charting and forecast visualisation rebuilt with Power BI as the rendering layer. Unlocked richer interactivity, custom measures, and enterprise export without maintaining a bespoke chart library.
Phase 5 — Launch
Expana brand goes live
Platform shipped under Expana branding with zero legacy customer disruption. New onboarding flow surfaced forecasting and dashboarding features to existing users during migration.

Markets — Red meat pricing tables

The Markets section handles highly structured, data-dense content — US red meat pricing sourced from USDA reports. Procurement professionals need to scan, compare, and export hundreds of line items efficiently. The redesign introduced a completely new table architecture.

🥩   Red Meat Prices — US | Boxed Beef Detail (31 July 2024)
US Red Meat Prices — Boxed Beef view with expandable categories (Ribs, Chucks), aggregate summary cards, and multi-select row comparison

Table design challenges

  • Information density — 12+ columns required. Solution: collapsible column groups with a Columns picker to show/hide sets, with procurement-relevant defaults pre-selected
  • Category navigation — Beef, Pork, Lamb & Veal, and Supply each need distinct sub-tabs (Boxed beef, Boneless beef, Imported beef, Cutter cow, Beef cutout, etc.) without full page reloads
  • Row comparison — checkboxes enable multi-row selection with a sticky comparison bar; selections persist across sub-tab switches within the same date view
  • Date navigation — single-day date picker with forward/back arrows enables day-by-day browsing without losing scroll position or selection state
  • Summary cards — aggregate Loads and Pounds totals pinned above the table give buyers instant headline figures without scanning the full dataset

Designing for Power BI from day one

Most product designers hand off to engineers and let the BI team figure out how charts should look. For Expana, that approach would have produced two disconnected visual languages — a polished web UI and a default-styled Power BI report that felt like a different product entirely. Instead, every design decision was made with Power BI's constraints and capabilities in mind from the start.

The rule was simple: if it can't be built in Power BI, it doesn't exist in the design. Every visual, colour, spacing decision, and layout choice was validated against what Power BI's embedded SDK could actually render — before a single pixel was finalised in Figma.

The design-to-Power BI workflow

Rather than treating Power BI as an afterthought, the design process was structured in five deliberate stages — each one producing an artifact that fed directly into the Power BI build.

🎯
01 · Question first
Define the single business question each report page must answer before touching Figma or Power BI
📐
02 · Canvas spec
Set canvas to 1664×936px. Design background layout as a PNG in Figma — imported as the bottom layer in Power BI
🎨
03 · Theme JSON
Export Expana colour tokens as a Power BI theme JSON file. Applied globally so every visual inherits brand colours automatically
📊
04 · Visual placement
Visuals placed precisely over background using the 3-30-300 layout rule: KPIs top, trends mid, granular data bottom
✅
05 · QA in embed
Every report QA'd in the embedded iframe context — not just Power BI Desktop — to catch spacing and clip issues

The 6 design principles applied to every Power BI report

🏗️
Principle 01
Background-first layout
All structural layout — panels, header bands, section dividers — is built as a Figma-designed PNG background image imported into Power BI. Visuals sit precisely on top. This avoids the "shape drift" problem where manually placed Power BI shapes misalign when the report evolves.
🎨
Principle 02
3–5 colour rule, always
Every report uses the Expana palette: 2 primary brand greens, 1 neutral, and 1 high-saturation accent reserved for the single most actionable metric per page. The same green is used for the same metric type across every report — users learn the colour language once and read any page faster.
📏
Principle 03
8-point grid & breathing room
Visual spacing follows an 8pt grid — 8px, 16px, 24px gaps. Power BI's canvas is 1664×936px (30% larger than default) to provide genuine breathing room between visuals. Dense reports are limited to 8–10 visuals per page; anything more gets its own drill-through page.
📖
Principle 04
3-30-300 layout rule
In 3 seconds, the user sees whether things are on track (KPI cards, top band). In 30 seconds, they understand the trend (time-series chart, mid section). In 300 seconds, they can investigate specific anomalies (granular table, bottom section). Every page is designed to answer all three in sequence.
🔤
Principle 05
Dynamic titles, not static labels
Chart titles and KPI labels use DAX-powered card visuals rather than static text boxes. When a user applies a slicer, the title updates to reflect the current selection — "Butter prices | France | Jan–Jun 2024" rather than a generic "Price chart". This reduces misinterpretation of filtered data.
♿
Principle 06
Colour-blindness safe by default
All multi-series charts use the Expana palette tested against deuteranopia and protanopia simulations. Line charts also use distinct stroke weights (solid, dashed, dotted) as a secondary encoding — so trend direction is readable in greyscale, in print, and for colour-blind users.

How the Expana theme JSON works

Rather than manually styling every visual in every report, the entire Expana visual language is encoded in a single Power BI theme JSON file. Once applied, every new chart automatically inherits the correct fonts, colours, backgrounds, and border radii — the same values used in the web platform's CSS tokens.

// expana-theme.json — applied to all Expana Power BI reports
{
  "name": "Expana",
  "dataColors": [
    "#1A5C3A",  // Primary green — main data series
    "#4CAF78",  // Accent green — secondary series
    "#00C853",  // Highlight — single actionable metric
    "#8FC9A8",  // Muted — tertiary / background series
    "#DDE8E2"   // Neutral — gridlines, borders
  ],
  "background": "#FFFFFF",
  "foreground": "#0D1F17",
  "tableAccent": "#1A5C3A",
  "visualStyles": {
    "*": { "*": {
      "fontFamily": [{ "value": "Inter, sans-serif" }],
      "background": [{ "transparency": 100 }],
      "border": [{ "show": false }]
    }},
    "lineChart": { "*": {
      "strokeWidth": [{ "value": 2 }],
      "markerSize": [{ "value": 0 }]  // clean line, no dots cluttering price charts
    }},
    "card": { "*": {
      "labelFontSize": [{ "value": 11 }],
      "labelColor": [{ "solid": { "color": "#5A6E63" }}]
    }}
  }
}

Design rules that kept web and Power BI in sync

🔁
One colour system, two outputs
Every colour was defined once in the Figma token library, then exported as both a CSS variable (for the web platform) and a hex value in the Power BI theme JSON. Updating a brand colour required a single change that propagated to both environments.
📐
Figma canvas matches Power BI canvas
Every Power BI report was designed at exactly 1664×936px in Figma before any Power BI build started. The background PNG exports at this resolution and imports without scaling artefacts. Visual placement coordinates from Figma translate 1:1 to Power BI's X/Y/W/H fields.
🚫
No pie charts, ever
Commodity price data is about magnitude and trend — not proportionality. Pie and donut charts were banned from the design system. Line charts, bar charts, KPI cards, and sparklines handle all use cases without introducing the angular misreading that pie charts cause.
💬
Tooltips as the detail layer
Rather than cluttering the canvas with data labels, custom report-page tooltips surface granular detail on hover. Designed as dedicated Power BI tooltip pages at 320×240px, styled to match the Expana card component — giving users depth without visual noise.
🏷️
DAX-driven dynamic titles
Static labels are a common Power BI mistake. Every chart title and KPI descriptor is a DAX measure that updates dynamically based on active slicers. When a buyer filters to "Butter | France | Q2 2024", every title on the page reflects that context — eliminating misreading risk.
📱
Mobile layout as a checkpoint
Every report was checked in Power BI's mobile layout view, even if mobile wasn't the primary use case. If key KPIs couldn't be rearranged into a readable vertical stack, the desktop layout was reconsidered — a lightweight constraint that consistently produced cleaner report structures.

Design system & Power BI alignment

A core deliverable alongside the product redesign was a component library that could be handed to developers and simultaneously used to specify Power BI report themes — ensuring visual consistency between the web platform and all embedded analytics views.

The decision to build around Power BI wasn't a technology constraint — it was a strategic choice. Expana's enterprise customers already lived in Power BI. By meeting them there, the analytics felt native to their workflow, not bolted on.
